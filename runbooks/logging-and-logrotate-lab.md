# Part 2 — Recognizing Rotated Logs

Several rotation naming patterns were visible:

```text
auth.log
auth.log.1
```

and:

```text
dmesg
dmesg.0
dmesg.1.gz
dmesg.2.gz
```

The general pattern is:

```text
application.log       → active log
application.log.1     → newest rotated log
application.log.2.gz  → older compressed log
application.log.3.gz  → even older compressed log
```

The active log normally has no numeric suffix.

The newest rotated log often uses `.1`.

Older logs may be compressed with gzip and use the `.gz` suffix.

## Permission Interpretation

```text
0644
```

means:

```text
owner: read and write
group: read only
other: read only
```

The log could therefore be modified by `root`, while other users could read it.

## Timestamp Fields

The `stat` output included:

```text
Access
Modify
Change
Birth
```

### Access

The last time the file content was read.

### Modify

The last time the file content was changed.

### Change

The last time file metadata or content changed.

### Birth

The file creation time, when supported by the filesystem.

## Inode

The inode uniquely identifies the filesystem object.

The inode value was recorded before rotation so the rotation method could be verified later.

# Part 6 — Creating the Logrotate Rule

A dedicated configuration file was created:

```text
/etc/logrotate.d/phoenix-heartbeat
```

Final configuration:

```conf
/var/log/phoenix-heartbeat.log {
    su root root
    daily
    rotate 7
    compress
    delaycompress
    missingok
    notifempty
    create 0644 root root
}
```

## Directive Explanation

### `su root root`

Runs the rotation for this rule using:

```text
user: root
group: root
```

This was required because `/var/log` was writable by the `syslog` group and logrotate considered the parent directory permissions unsafe without an explicit execution identity.

### `daily`

The log becomes eligible for rotation once per day.

This does not mean that logrotate runs continuously.

It means that when logrotate runs, it checks whether at least one daily interval has passed.

### `rotate 7`

Keeps up to seven previous rotated copies.

Older copies are removed when the retention limit is exceeded.

### `compress`

Compresses older rotated logs using gzip.

### `delaycompress`

Delays compression by one rotation cycle.

The newest rotated file remains uncompressed as `.1`.

Older rotated files become `.gz`.

### `missingok`

Does not report an error if the log file does not exist.

### `notifempty`

Does not rotate an empty log file.

### `create 0644 root root`

Creates a new active log after rotation with:

```text
mode: 0644
owner: root
group: root
```

# Part 7 — Configuration Validation

The rule was tested in debug mode:

```bash
sudo logrotate -d \
  /etc/logrotate.d/phoenix-heartbeat
```

The `-d` option:

- reads the configuration;
- evaluates the rule;
- prints diagnostic information;
- does not rotate or modify files.

## Initial Validation Error

The first test returned:

```text
error: skipping "/var/log/phoenix-heartbeat.log"
because parent directory has insecure permissions
```

The parent directory was:

```text
drwxrwxr-x root syslog /var/log
```

Because the directory was group-writable by a group other than `root`, logrotate required an explicit `su` directive.

The rule was corrected by adding:

```conf
su root root
```

## Successful Validation

The second debug test returned:

```text
rotating pattern: /var/log/phoenix-heartbeat.log after 1 days
empty log files are not rotated
log does not need rotating
```

This confirmed:

- the syntax was valid;
- the permissions issue was resolved;
- the rule was successfully evaluated;
- no normal rotation was required at that moment.

# Part 8 — Forced Rotation

A controlled forced rotation was performed:

```bash
sudo logrotate -f \
  /etc/logrotate.d/phoenix-heartbeat
```

The `-f` option forces rotation even if the configured time interval has not passed.

The resulting files were checked with:

```bash
ls -lah /var/log/phoenix-heartbeat*
```

Result:

```text
-rw-r--r-- root root    0 phoenix-heartbeat.log
-rw-r--r-- root root  27K phoenix-heartbeat.log.1
```

This confirmed that logrotate:

1. renamed the active log to `.1`;
2. created a new empty active log;
3. preserved the configured ownership and permissions.

---

# Part 9 — Verifying Inode Replacement

The active and rotated files were compared:

```bash
sudo stat -c '%n inode=%i size=%s' \
  /var/log/phoenix-heartbeat.log \
  /var/log/phoenix-heartbeat.log.1
```

Result:

```text
/var/log/phoenix-heartbeat.log inode=786745 size=0
/var/log/phoenix-heartbeat.log.1 inode=786837 size=27136
```

Before rotation, the active log used inode:

```text
786837
```

After rotation:

```text
phoenix-heartbeat.log.1 → inode 786837
phoenix-heartbeat.log   → inode 786745
```

This proved that the original file was renamed and a new filesystem object was created for the active log.

This is different from a `copytruncate` rotation, where the existing active file normally retains its inode.

# Part 15 — Logrotate State File

The logrotate state was checked:

```bash
sudo grep "phoenix-heartbeat" \
  /var/lib/logrotate/status
```

Result:

```text
"/var/log/phoenix-heartbeat.log" 2026-7-22-19:31:15
```

The state file records when a specific log was last rotated.

This allows logrotate to determine whether a rule using `daily`, `weekly`, or another interval is eligible for another rotation.

## Important Distinction

### `logrotate.timer`

Controls when the logrotate service is started.

### `daily`

Defines the minimum rotation interval for the individual rule.

### State file

Records when each log was last rotated.

### `-f`

Forces rotation and ignores the normal interval decision.

# Troubleshooting

## Error: Insecure Parent Directory Permissions

Example:

```text
error: skipping log because parent directory has insecure permissions
```

Resolution:

Add an explicit `su` directive:

```conf
su root root
```

Do not change the permissions of `/var/log` without first understanding the system logging design.

## Log Does Not Rotate

Check the rule:

```bash
sudo logrotate -d \
  /etc/logrotate.d/phoenix-heartbeat
```

Check the state file:

```bash
sudo grep "phoenix-heartbeat" \
  /var/lib/logrotate/status
```

Possible reasons:

- the minimum interval has not passed;
- the log is empty and `notifempty` is enabled;
- the file does not exist and `missingok` is enabled;
- the state file records a recent rotation;
- the rule contains a syntax or permission error.

## New Active Log Is Not Created

Verify that the rule contains:

```conf
create 0644 root root
```

Also verify that the executing user has permission to create files in the parent directory.

## Service Stops Logging After Rotation

Possible causes:

- the service keeps an old file descriptor open;
- the service requires a reload signal;
- the new log has incorrect ownership;
- the new log has incorrect permissions;
- the application does not reopen its log automatically.

Possible solutions may include:

```text
postrotate
systemctl reload
systemctl kill with an appropriate signal
copytruncate
```

The correct method depends on the application.

`copytruncate` should not be used automatically because it can introduce a small risk of lost log entries during copying and truncation.

## Compressed Log Cannot Be Read with `cat`

Use gzip-aware tools:

```bash
zcat file.log.gz
```

```bash
zless file.log.gz
```

```bash
zgrep "pattern" file.log.gz
```

# Recovery and Removal

## Validate Before Changing Anything

```bash
sudo logrotate -d \
  /etc/logrotate.d/phoenix-heartbeat
```

## Remove the Dedicated Rule

Only when the custom logging configuration is no longer required:

```bash
sudo rm /etc/logrotate.d/phoenix-heartbeat
```

Removing the rule does not automatically remove existing log files.

## Do Not Remove the Global Timer

The following system timer is shared by many system logs:

```text
logrotate.timer
```

It should not be disabled merely to remove one custom log rule.

## Manually Rotate for Testing

```bash
sudo logrotate -f \
  /etc/logrotate.d/phoenix-heartbeat
```

Forced rotation should be used carefully in production because repeated execution can consume retention slots and remove older logs sooner than expected.

# Security Considerations

- Log files may contain usernames, source IP addresses, service names, paths, and operational details.
- Authentication logs should not normally be world-readable.
- Log ownership should match the application or system logging model.
- Do not grant broad write access to `/var/log`.
- Do not change `/var/log` ownership to solve a single application issue.
- Use dedicated logrotate rules for custom application logs.
- Validate configurations before forcing rotation.
- Protect compressed archives with appropriate ownership and permissions.
- Include important logs in backup and incident-response planning.
- Retention periods should reflect operational, security, and legal requirements.
- Sensitive secrets, private keys, tokens, and passwords must never be written to logs.

---

# Backup and Recovery Considerations

Log rotation is not a backup system.

Rotation provides:

- local retention;
- disk-space control;
- historical troubleshooting data.

It does not provide protection against:

- disk failure;
- filesystem corruption;
- malicious deletion;
- host compromise;
- complete VM loss.

Important logs may also require:

- remote log forwarding;
- centralized logging;
- protected backup storage;
- monitoring and alerting;
- defined retention policies.

# Lessons Learned

- Linux uses both systemd journal storage and traditional text log files.
- `/var/log` contains logs with different owners, groups, and access levels.
- Rotated logs use numbered suffixes.
- Older logs are commonly compressed.
- File metadata and inode values help explain rotation behavior.
- Logrotate rules can be stored independently in `/etc/logrotate.d/`.
- Debug mode validates a rule without modifying files.
- Parent-directory security checks can require an explicit `su` directive.
- `create` produces a new active log after rotation.
- `delaycompress` keeps the newest rotated log uncompressed.
- `zcat` can read gzip-compressed logs directly.
- `logrotate.timer` starts the automatic rotation process.
- `logrotate.service` is a normal oneshot service.
- The state file tracks the last rotation time for each log.
- Rotation, retention, compression, and backup are separate concepts.
- A successful rotation must also verify that the application continues logging.
